# Dino AI 新老 App 用户迁移方案

> 版本：v1.0 ｜ 更新日期：2026-07-09
> 目标：明确老 App 用户是否迁移至新 App，并给出迁移 / 不迁移的优劣势对比与落地建议。

---

## 一、背景与基础信息

### 1.1 新老 App 对比

| 维度 | 老 App | 新 App | 差异点 |
| --- | --- | --- | --- |
| 登录注册方式 | 手机号、kakao、谷歌邮箱、Facebook、AppID | 手机号、kakao、谷歌邮箱、Facebook、AppID | 完全一致 ✅ |
| 会员类型 | 月包、年包 | 周包、月包、年包 | 新增「周包」 |
| 覆盖国家 | 韩国、沙特、马来 | 韩国、沙特、马来、印尼、越南 | 新增「印尼、越南」 |

### 1.2 关键结论（先看这里）

- **登录方式完全一致** → 账号体系可无缝对齐，迁移的最大技术障碍（账号匹配）风险很低。
- **会员类型仅新增周包** → 老用户的月包 / 年包权益可 1:1 映射到新 App，不存在权益缩水。
- **国家仅做增量扩张（印尼、越南）** → 老 App 覆盖的三国用户在新 App 均被覆盖，不存在「迁过去无法服务」的地区。

综合来看，**本次迁移属于「低风险、高收益」的场景**，推荐执行迁移（详见第四章建议）。

---

## 二、迁移方案（推荐路径）

### 2.1 迁移的优势

| 类别 | 优势 | 说明 |
| --- | --- | --- |
| 用户资产 | 保留存量用户 | 老 App 用户直接转化为新 App 用户，避免重新拉新的高成本 |
| 用户资产 | 保留付费用户 | 月包 / 年包用户及其订阅关系延续，降低付费流失 |
| 产品体验 | 统一体验 | 用户集中在新 App，享受新功能（如周包）与后续迭代 |
| 运营效率 | 单端维护 | 逐步下线老 App，减少双端并行的开发、客服、运营成本 |
| 数据价值 | 数据归一 | 用户行为 / 付费数据统一沉淀，便于分析与增长决策 |
| 商业扩张 | 承接新市场 | 新 App 已覆盖印尼、越南，迁移后可复用统一的增长与投放体系 |
| 会员升级 | 周包引流 | 周包降低付费门槛，可对老用户做转化 / 复购的二次运营 |

### 2.2 迁移的劣势 / 风险

| 类别 | 劣势 / 风险 | 缓解措施 |
| --- | --- | --- |
| 技术 | 账号匹配失败 | 以登录方式为唯一键做映射；建立兜底人工核对通道 |
| 技术 | 订阅态迁移复杂 | 苹果 / Google / 第三方支付的订阅需按平台规则迁移或重建，提前与支付渠道确认 |
| 技术 | 进度 / 数据丢失 | 迁移前做全量备份 + 抽样对账（≥98% 一致率）|
| 用户 | 迁移感知打断 | 提供「一键迁移」引导页，减少操作步骤 |
| 用户 | 老用户流失 | 迁移期给予迁移激励（如赠送周包 / 续费折扣）|
| 客服 | 咨询量上升 | 提前准备 FAQ、客服 SOP，设置迁移专属客服入口 |
| 合规 | 跨国数据合规 | 韩国 / 沙特 / 马来的数据与隐私合规需分区域核对 |
| 财务 | 重复购买风险 | 老端订阅用户在新端误重复购买，需做订阅去重校验（目标 < 0.5%）|

## 三、不迁移方案（备选路径）

即：老 App 与新 App 并行运营，不强制用户迁移。

### 3.1 不迁移的优势

| 类别 | 优势 | 说明 |
| --- | --- | --- |
| 风险 | 零迁移风险 | 无账号匹配、订阅迁移、数据丢失等技术风险 |
| 用户 | 无打扰 | 老用户维持原有体验，不产生迁移流失 |
| 节奏 | 灰度可控 | 新 App 可专注新市场（印尼、越南）快速验证 |
| 成本 | 短期投入低 | 无需投入迁移工具、对账、客服 SOP 等一次性成本 |

### 3.2 不迁移的劣势

| 类别 | 劣势 | 说明 |
| --- | --- | --- |
| 成本 | 双端长期维护 | 开发、测试、客服、运营长期双份成本，边际效率低 |
| 数据 | 数据割裂 | 用户 / 付费数据分散在两端，难以形成统一增长视图 |
| 体验 | 体验分裂 | 新功能（周包等）无法惠及老用户，老用户体验逐渐落后 |
| 增长 | 存量无法激活 | 老 App 存量用户无法导入新市场增长体系 |
| 品牌 | 认知混乱 | 同品牌双 App 易造成用户与市场认知混乱 |
| 长期 | 技术债累积 | 老 App 维护成本随时间上升，最终仍需迁移，越晚越贵 |

## 四、结论与建议

### 4.1 决策建议

**推荐执行「迁移方案」，采用「渐进式灰度迁移」而非强制一刀切。**

核心理由：
1. 登录方式一致 + 会员权益可 1:1 映射 + 国家全覆盖 → 迁移的核心风险（账号 / 权益 / 地区）都很低。
2. 双端并行的长期成本远高于一次性迁移成本。
3. 新 App 已具备承接老用户 + 拓展新市场的能力，迁移能让数据与增长归一。

### 4.2 迁移策略（渐进式）

| 阶段 | 目标 | 动作 |
| --- | --- | --- |
| 准备期 | 打通能力 | 账号映射打通、订阅迁移方案确认、对账工具、客服 SOP |
| 灰度期 | 小范围验证 | 选韩国 5% 付费用户做迁移引导，监控登录 / 恢复率 |
| 扩大期 | 覆盖核心区 | 扩展至沙特、马来，监控客诉与订阅异常 |
| 全量期 | 全量迁移 | 全量老用户迁移引导，老 App 进入维护模式 |
| 复盘期 | 评估收尾 | 迁移完成率、新 App 留存、老 App 下线时间点评估 |

### 4.3 成功指标（参考）

| 指标 | 目标 |
| --- | --- |
| 账号匹配成功率 | ≥ 99% |
| 进度 / 订阅恢复一致率 | ≥ 98%（抽样对账）|
| 订阅相关客诉 | 周环比不上升 |
| 老端订阅用户新端重复购买率 | < 0.5% |
| 迁移用户 D7 留存 | ≥ 自然新用户同期 |
